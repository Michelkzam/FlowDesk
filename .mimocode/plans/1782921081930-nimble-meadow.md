# Plano: Criar API Route `/api/tickets/[id]/close`

## Contexto

O frontend (`TicketDetail.jsx`) já consome uma lógica de finalização via `handleFinalize()` (linha 317) que usa `updateMutation.mutate()` diretamente no Supabase. Porém, não existe uma API route server-side para fechar tickets — a lógica atual é toda client-side, sem validação server-side de permissões RBAC.

A API route `/api/tickets/[id]/close` centralizará a lógica de fechamento com:
- Validação server-side de RBAC (`tickets:close`)
- Criação de mensagem de sistema com a solução
- Registro de audit log
- Atualização atômica do ticket (status + closed_date)
- Broadcast realtime

## Arquivo a Criar

```
api/tickets/[id]/close.js
```

## Implementação

### Endpoint: `POST /api/tickets/:id/close`

**Request Body:**
```json
{
  "status": "resolved" | "closed",   // obrigatório
  "category_name": "string",          // opcional
  "solution": "string"                // obrigatório (mínimo 15 caracteres)
}
```

**Response 201:**
```json
{
  "success": true,
  "ticket": { "id", "status", "closed_date" }
}
```

### Lógica (seguindo padrão de `messages.js` e `transfer.js`):

1. **Auth**: Verificar Bearer token via `supabase.auth.getUser(token)`
2. **RBAC**: Verificar se usuário tem `tickets:close` via `supabaseAdmin.rpc('usuario_tem_permissao', ...)`
3. **Validações**:
   - Ticket existe
   - Ticket não está já fechado (`status !== 'closed'`)
   - `solution` obrigatório com mín. 15 caracteres
   - `status` deve ser `"resolved"` ou `"closed"`
4. **Mensagem de sistema**: Inserir em `ticket_messages` com prefixo `[Solução]`
5. **Atualizar ticket**: `status` + `closed_date` + `updated_at`
6. **Audit log**: Registrar `ticket_closed` com old_value/new_value
7. **Retornar** ticket atualizado

### Arquivos Existentes para Referência (padrão a seguir):
- `api/tickets/[id]/messages.js` — auth + RBAC pattern
- `api/tickets/[id]/transfer.js` — validações + audit log pattern
- `api/cron/auto-close.js` — lógica de fechamento automático (system message + status update)

### Frontend (não alterado):
- `TicketDetail.jsx` pode ser atualizado depois para chamar esta API route em vez de `db.entities.Ticket.update()` direto
- A ResoluçãoModal já coleta `category` e `solution` que são os campos aceitos pela API
